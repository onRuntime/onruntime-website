## 定义

框架就像是我们在 onRuntime 的预制工具箱。与其为每个项目从头开始构建一切（想象一下，造一栋房子要自己造钉子！），我们使用这些结构，它们为我们提供了良好的起点和稳固的约定。

## 大问题：框架还是库？

我们常常收到客户的这个问题，所以事情是这样的：

- **库**就像是借用一个特定任务的具体工具。你掌控一切，决定何时使用它。
- **框架**就像是进入一个配备齐全的车间，遵循它自己的规则。它告诉你怎么工作。

有句名言很好地总结了这一点：“你调用一个库，但一个框架调用你。”

## 我们喜欢使用的框架

### 前端部分
- **React**：技术上是一个库，但周围的生态系统（React Router, Redux 等）几乎让它成为一个框架。我们在用户界面方面的首选。
- **Next.js**：我们对复杂网站的最爱！我们用它重建了我们自己的 onRuntime.com 网站。
- **React Native**：对于我们客户的移动应用，它是我们的首选。我们在 Tonight Pass 中使用它，效果很好。

### 后端部分
- **NestJS**：我们完全是粉丝！它结构清晰，模块化，节省了我们在大型应用上的大量时间。
- **Express**：比 NestJS 更轻，我们用它来构建简单的 API 或 MVP。
- **Prisma**：虽然不能算是一个完整的后端框架，但它是我们管理数据库的最爱 ORM。

### 全栈
- **Next.js**（又来了！）：我们越来越多地使用它作为全栈解决方案，得益于 API 路由。
- **Redwood.js**：我们在几个项目上测试过它，前景不错！有点像 Ruby on Rails，但适用于 JS 生态系统。

## 为什么我们不从头开始编写所有代码

- **我们能快很多**：不再有别人已经解决了一千次的问题
- **更干净的代码**：框架约定强迫我们保持纪律
- **安全性**：流行框架经过成千上万开发者的审查，漏洞被追踪
- **更简单的入职**：当我们将新开发者融入团队时，如果他们已经了解这个框架，能迅速进入状态

## 我们何时选择使用（或不使用）框架

在 onRuntime，我们喜欢框架，但我们并不盲目使用。对于超具体的微服务，我们有时更喜欢轻量级的方法，仅使用几个库。例如，对于我们的 Kartrak 应用，我们选择了更简约的方法，因为性能至关重要。

但老实说，对于大多数客户项目，时间到市场很重要，一个好的框架能节省大量时间，选择显而易见！
